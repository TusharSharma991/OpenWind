## Executive summary

OpenWind is a **genuinely working, multi-tenant, workflow-native business platform** — not vaporware, but also not the seven-product suite the README advertises. The honest core is this: a stateful object (ticket, deal, invoice, leave request) moving through a configurable state machine, and **that core is real and production-grade**. Three engines (Entity, Workflow, Automation) are implemented with substance; a generic HTTP API serves any entity type; multi-tenancy is enforced at the Postgres layer via Row-Level Security (not app-layer `WHERE` clauses); and there are **two fully-built React frontends** — an admin console with a visual state-machine builder, custom-field editor, and automation wizard, plus a customer portal for submitting and tracking records. The seven "modules" exist, but they are **thin seed-SQL templates** (mostly one entity type each, almost no automations, zero sample records) that _prove the config-first thesis_ rather than ship deep products. The advertised differentiators that are **not built yet**: AI features (the `ai` package is a bare Anthropic client), connectors (Slack/email/Stripe/WhatsApp are SDK interfaces with zero implementations), the plugin runtime, and — critically — **end-to-end notification delivery** (notifications enqueue but the `notify` automation action is a stub and the Novu worker is unwired). Net: the _platform_ is ~80% of a compelling product; the _applications and integrations layered on it_ are early. Note also that the planning docs are **stale** — VISION.md/CLAUDE.md claim module track "2B" is "0% done," but all seven module seeds are in fact present and functional. Read the code, not the docs.

## What the platform can do (the config leverage)

A non-engineer, working entirely through the admin UI or seed SQL — **no code, no migration, no deploy** — can today:

- **Define a new entity type** with any of 15 field types (text, currency, date, enum, multi-enum, user_ref, entity_ref, file/files, formula, lookup, …), each with validation rules. (`entity-types/detail.tsx`, `field-types.ts`)
- **Add per-tenant custom fields** to an existing entity, instantly, stored as JSONB with no schema migration. (`engine.ts addEntityField`)
- **Draw a workflow** visually — states (with colours, terminal flags, **SLA hours**), and transitions with **role guards, required comments, and required fields** — on a ReactFlow canvas. (`workflows/detail.tsx`)
- **Build automation rules** via a 4-step wizard: pick a trigger, compose a nested AND/OR/NOT condition tree over field values, attach actions. (`automations/wizard/*`) — _with the caveat that only `set_field`, `transition`, and `webhook` actually execute._
- **Configure SLA timers** that fire reliably through BullMQ and emit a breach event. (`sla-scheduler.ts`)
- **Fire outbound webhooks** on any state change — SSRF-protected, safe to point at external systems today. (`actions/webhook.ts`)
- **Fork a whole module** (entity type + workflow) into their tenant in one click. (`modules.tsx`)
- **Customize list/detail/form layouts** per entity type, and save personal filtered views. (`admin/view-configs`, `saved-views`)
- **Export any dataset** to CSV/XLSX/PDF with role-based PII redaction, and **query the audit trail**.

Genuinely **config-driven**, no code: entity types, fields, workflows, transitions, guards, SLAs, automation rules (for the 3 working actions), view layouts, module install. **Needs code today**: notifications/escalation delivery, any connector, AI, plugins, the four unimplemented automation actions, and the two unimplemented triggers.

## Easy wins (ranked by leverage ÷ effort)

1. **Wire `notify` + the Novu delivery worker — close the notification loop.** _Highest leverage, smallest surface._ The enqueue API, per-user channel preferences, and template cache already exist (`packages/notifications`); only `actions/notify.ts` (a one-function stub) and a worker consumer are missing. This single change activates: notification automations, **SLA escalation** (which is _designed_ to be "automation reacts to `sla_breached` → notify"), and the helpdesk/approval flows that currently transition silently. Without it, half the platform's promised value is dark.
2. **Implement `assign` and `create_entity` automation actions.** Both reuse engines that already exist — `assign` is a `set_field` on a `user_ref`; `create_entity` is a call into `entity-engine.createEntity`. A day or two each. `create_entity` also unblocks the documented **parallel-approval-via-related-entities** pattern without building a new engine.
3. **Enrich the six bare module seeds — pure SQL, zero code.** Add SLAs to states and a handful of automations (invoice→auto-overdue on due date once `schedule.cron` lands; leave-request→notify approver; reimbursement→notify on submit). This is exactly the config-first thesis and turns thin templates into demoable products. Pair with fixing the **helpdesk duplicate-entity-type bug**.
4. **Ship one real connector (email-in or Slack) on the existing SDK.** The README sells connectors; the SDK contract is done (`connector-sdk/types.ts`). Email-to-ticket is the canonical helpdesk closer and a single connector file. High buyer-visible payoff.
5. **A reporting/dashboard surface.** All the data is already queryable (full-text search, event log, export, instance counts). A reports page beyond today's KPI cards is mostly frontend assembly over live endpoints — no backend work.
6. **A new module nobody's asked for yet (e.g. Asset Management, Contracts) as seed SQL only.** Proven cheap — the install runner and registry work — and a fast way to demonstrate "ship an app as configuration."
7. **First AI assist using the already-installed Anthropic client.** Ticket classification or draft-reply (review-before-action, per VISION) is a contained, single-call feature on `packages/ai` that doesn't need the full RAG stack.

## Gaps & deferred (can't-do-today)

- **Notification delivery is not end-to-end** — the most important gap; everything user-facing that "should notify" currently doesn't.
- **No working connectors at all** — Slack/email/Stripe/WhatsApp are interfaces. No inbound channels (email-to-ticket), no payment capture, no chat.
- **AI features entirely absent** despite prominent README/tech-stack billing.
- **Plugin runtime absent** — no third-party extensibility yet (Phase 3).
- **Parallel/multi-approver approval deferred** — only sequential approval; blocks many real HR/procurement/finance flows.
- **No time-based or field-change automations** — `schedule.cron` and `field.changed` triggers are typed but never fire; only SLA-based timing exists.
- **No user / role / org-management UI** — roles are read-only from Zitadel; admins can't manage their own users in-product. Settings is appearance-only.
- **No analytics/BI** beyond KPI cards; **no billing/payments** despite the Invoicing module; **no mobile**.
- **Planning docs are stale** (VISION/CLAUDE claim modules "0%"; they're built) — a governance gap that will mislead anyone roadmapping from docs alone.

## Differentiators (framed for a buyer)

- **Config-is-the-product, and it's proven, not promised.** Seven modules exist as pure seed SQL with zero per-module TypeScript — a customer genuinely adds a field, a state, or a rule without waiting for a developer or a deploy. The visual workflow builder and automation wizard make that real for non-engineers. This is the Zoho/ServiceNow pitch, but **open-source and self-hostable** (AGPL).
- **Tenant isolation at the database, not the application.** RLS via `app.tenant_id` GUC means a forgotten `WHERE` clause still can't leak across tenants — a materially stronger security story than typical app-layer-scoped SaaS. (`0001_rls_and_tenancy.sql`)
- **Everything is an event, and the event log is immutable.** Every transition is appended to `workflow_events`; every admin action to `admin_audit_log` with before/after snapshots and PII redaction. Inspectable, exportable, auditable by design — compelling for regulated buyers.
- **Security posture beyond the demo bar:** SSRF-hardened + DNS-pinned outbound webhooks, async AV scanning of every upload, presigned-only file access, GDPR tenant-purge worker, scoped API keys. These are usually afterthoughts; here they're shipped.
- **One unified data layer.** A generic entity API and shared engines mean a ticket, a deal, and an invoice are the same primitive — cross-object relations and a single search/export/audit surface, versus the disconnected-tools sprawl the product is positioned against.

The flip side a buyer must hear: integrations, AI, and notification delivery — the things that make a platform _feel alive day-to-day_ — are the least-built parts. OpenWind today is a superb **engine and back office** waiting for its **connective tissue**.

## Suggested 3-horizon roadmap baseline

**Now (close the loop — make what exists actually deliver):**

- Wire `notify` action + Novu delivery worker; turn on SLA escalation as a seeded automation. (#1)
- Implement `assign` and `create_entity` automation actions. (#2)
- Fix the helpdesk duplicate-entity-type bug; enrich the six bare module seeds with SLAs + starter automations. (#3)
- Refresh VISION/CLAUDE/roadmap docs to match the built reality (modules are done, not 0%).

**Next (connective tissue — make it feel like a platform):**

- First-party connectors v1: email-in→ticket and Slack notify, on the existing SDK. (#4)
- `schedule.cron` and `field.changed` triggers (unlocks recurring billing, auto-overdue, change-driven rules).
- Reporting/dashboard surface over existing search/export/event data. (#5)
- User/role/org-management admin UI (own your tenant without touching Zitadel).
- Sequential-to-parallel approval (now cheap once `create_entity` exists — related-approval-entities pattern).
- First AI assist (ticket classification / draft reply, review-before-action). (#7)

**Later (extensibility & depth — become a foundation others build on):**

- Plugin runtime (loader, registry, slot/hook/job dispatch) for third-party extensions.
- Billing/payments (Stripe connector) behind the Invoicing module.
- Full RAG/automation-generation AI assistants.
- Analytics/BI, white-label/theming, sector packages, and a mobile/responsive surface.
- Connector marketplace (per-tenant install) once 2-3 first-party connectors validate the SDK.
